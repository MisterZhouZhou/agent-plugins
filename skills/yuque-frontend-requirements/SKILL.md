# 语雀前端需求提炼

此技能用于把语雀需求文档转换为两类产物：

1. 原始清洗文档：通过语雀 API 拉取正文，去掉冗余 HTML、样式和无关包裹结构，输出可读 Markdown。
2. 前端技术文档：基于清洗后的 Markdown，按固定模板整理为前端可执行的实现需求文档。

脚本只负责第 1 步的“拉取和清洗”。第 2 步的“提炼和成文”由 agent 按模板完成。

## 何时使用

- 用户给出语雀文档，希望整理成前端可开发的技术需求文档。
- 用户要求把语雀 PRD、交互说明、页面需求、业务方案转成前端实现文档或开发提示词。
- 语雀文档正文混有 HTML、样式标签、图片包裹、复制痕迹，直接阅读成本高。
- 用户特别强调要保留原始需求标题、字段映射、状态枚举、接口说明、参考代码路径。

## 不适用场景

- 用户只想搜索语雀知识库里的资料，而不是处理单篇文档。此时更适合使用 `yuque-kb-search`。
- 用户提供的不是语雀文档，而是本地 Markdown、飞书文档、Notion、截图或 PDF。
- 用户要的是产品总结、运营摘要、汇报材料，而不是前端实现需求。

## 必要输入

- 语雀文档链接，或 `repo namespace + doc slug`
- 可读取该文档的语雀 Token

可选输入：

- 参考代码路径
- 组件写法约束
- 接口调用示例
- 字典获取方式
- 既有页面或模块实现锚点

如果文档是私有知识库且当前没有可用 Token，先向用户索要 Token 再继续。

## 资源使用说明

- 先按本文件执行主流程。
- 需要查看接口请求格式、`404` 排查项时，读取 `references/yuque_api_usage.md`。
- 需要向人解释如何使用此技能时，读取 `references/usage_guide.md`。
- 拉取和清洗语雀正文时，使用 `scripts/yuque_to_markdown.py`。
- 输出技术文档时，使用 `assets/frontend_requirement_template.md` 作为骨架。

## 标准工作流

### 1. 识别文档身份

- 优先从语雀链接解析 `repo namespace` 和 `doc slug`。
- 支持的链接格式包括：
  - `https://www.yuque.com/owner/repo/doc-slug`
  - `https://tenant.yuque.com/owner/repo/doc-slug`
- 如果链接无法稳定解析，再要求用户补充 `repo namespace` 和 `doc slug`。

### 2. 确认访问凭证

- 优先使用显式传入的 `--token`。
- 如果执行环境已有 `YUQUE_TOKEN`，可直接复用。
- 没有 Token 时不要继续请求语雀 API。

### 3. 拉取并清洗原文档

在项目根目录下优先执行：

```bash
python3 skills/yuque-frontend-requirements/scripts/yuque_to_markdown.py \
  --url "https://www.yuque.com/owner/repo/doc-slug" \
  --output-dir outputs
```

如果链接不稳定或无法解析，改用：

```bash
python3 skills/yuque-frontend-requirements/scripts/yuque_to_markdown.py \
  --repo-namespace "owner/repo" \
  --doc-slug "doc-slug" \
  --output-dir outputs
```

补充约束：

- 通过 `--token` 或环境变量 `YUQUE_TOKEN` 传入 Token。
- 若用户没有指定输出路径，优先写入项目根目录下的 `outputs/`。
- 原始清洗结果命名规则为 `{title}-原文档.md`。
- 如需自定义后缀，可通过 `--output-label` 传入。

### 4. 阅读清洗结果，只提炼前端实现信息

优先提炼以下内容：

- 页面、路由、Tab、模块布局、展示区域
- 表单、筛选区、输入框、下拉框、时间选择器、弹窗、抽屉、表格、分页
- 字段定义、字段映射、默认值、占位文案、是否必填、是否禁用、状态枚举
- 按钮行为、查询重置、跳转查看详情、联动逻辑、显隐逻辑、查看态和空态
- 本地下拉选项、字典下拉选项、字典类型、接口参数映射
- 列表请求、详情请求、分页结构、请求参数、响应字段、字段展示映射
- 加载中、空数据、失败兜底、权限控制、异常提示
- 上传、预览、导出、复制、长图展示、PDF 或链接打开等前端行为
- 用户额外提供的参考代码路径或已有页面实现方式

不要提炼以下内容：

- 与前端实现无关的业务背景铺垫
- 没有落到页面或交互上的组织流程描述
- 源文档不存在的接口字段、状态枚举、条件分支

### 5. 按模板产出技术文档

以 `assets/frontend_requirement_template.md` 为骨架，新建 `{title}-技术文档.md`，并按以下规则填写：

- `# 参考上下文` 只承接用户额外明确提供的上下文，例如参考代码路径、组件写法、字典获取方式、接口调用示例。
- 用户没有提供额外上下文时，`# 参考上下文` 填写 `暂无`。
- 不要把原始语雀文档摘要塞进 `# 参考上下文`。
- `# 功能需求` 下的二级标题尽量复用原始需求标题，不要统一写成“需求描述”。
- 每个原始需求标题下输出的必须是“可落地的前端实现要求”，而不是原文复述。
- 对列表类需求，优先拆成：界面结构、筛选条件、按钮行为、表格字段、字段映射、详情跳转、分页逻辑。
- 对表单类需求，优先拆成：字段、校验、默认值、交互、提交流程、成功失败反馈。
- 对展示类需求，优先拆成：显隐条件、展示样式、交互入口、异常兜底、空态处理。
- 如果用户已经提供接口文档，则在 `## 接口描述` 中按模板整理；只提供部分接口时，仅填写已知部分，其余写 `暂无`。
- 如果用户提供了“字段映射”“状态枚举”“字典类型”“请求示例”等，优先保留原始命名，不自行改写。
- 如果某个结论无法从文档中确认，显式写明“待确认”或“暂无”，不要推断。
- 当某个编号条目下需要继续细分场景、类型或分支逻辑时，使用“编号项 + 次级短横线列表”的形式表达。
- 保留文末 `## 工作流程` 的协作约束，不删除、不改写语义。

### 6. 交付时的输出要求

- 最终交付物应只包含 Markdown。
- 如果用户要求文件落盘，默认输出两份文件：
  - `{title}-原文档.md`
  - `{title}-技术文档.md`
- 如果用户只要求在对话中返回内容，也要先给技术文档，再按需附原文档路径或摘要。
- 回答中应明确哪些内容来自原文，哪些位置因为信息缺失被标记为 `暂无` 或 `待确认`。

## 输出质量要求

- 保留原业务意图，但改写成前端可执行语言。
- 按原始需求标题分组，不随意重组主题。
- 缺失上下文、接口、字段、状态时明确写 `暂无`。
- 对模糊项使用 `待确认`，不要补猜。
- 文档应能直接作为“先确认需求，再拆任务，再开发”的输入材料。

## 常见失败与处理

- 语雀 API 返回 `404`：
  - 检查 Token 是否有该知识库访问权限。
  - 检查 `repo namespace` 和 `doc slug` 是否完全正确。
  - 检查请求是否带了 `raw=1`。
  - 检查请求是否带了 `User-Agent`。
- 返回 `403`：
  - 优先判断 Token 权限不足，或文档属于私有知识库。
- 能拉到原文但技术文档信息大量缺失：
  - 这是源文档信息不完整，不是技能失败。应明确写 `暂无` 或 `待确认`。
