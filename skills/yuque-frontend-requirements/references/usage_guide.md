# 语雀前端需求提炼 Skill 使用说明

## 这是什么

这个 Skill 用来把单篇语雀文档整理成适合前端开发继续使用的材料，通常会产出两份内容：

1. 原始清洗文档：把语雀正文里的 HTML、样式和无关包裹结构清洗成 Markdown。
2. 前端技术文档：基于清洗后的内容，按固定模板整理成前端可执行的实现需求文档。

其中第 1 份内容由脚本自动生成，第 2 份内容由 agent 按模板整理生成。

## 适合的场景

- 产品、运营或业务同学给了一篇语雀 PRD，希望快速转成前端技术文档。
- 语雀文档里混有大量 HTML 标签、样式内容或复制痕迹，直接阅读和拆解成本高。
- 需要保留原始需求标题、接口信息、字段映射、状态枚举、参考代码路径。
- 需要把语雀需求进一步整理成前端开发提示词或实现清单。

## 不适合的场景

- 你要搜索整个语雀知识库，而不是处理单篇文档。
- 你给的是飞书、Notion、本地 Markdown、截图或 PDF。
- 你要的是业务摘要、汇报材料或产品总结，而不是前端实现文档。

## 使用前准备

请准备以下信息：

1. 语雀文档链接，或 `repo namespace + doc slug`
2. 语雀 Token
3. 如果有的话，再补充参考上下文

Token 要求：

- 在语雀个人设置中创建
- 至少具备读取权限
- 如果文档是私有知识库，Token 对应账号必须能访问该文档

## 推荐提供的信息

如果希望输出结果更贴近你的前端项目，建议一并补充：

- 参考代码路径
- 参考页面
- 参考组件写法
- 列表请求示例
- 详情接口示例
- 字典获取方式
- 页面跳转参考

这些内容会进入最终文档的 `# 参考上下文`，方便后续开发继续沿用现有实现方式。

## 最简单的用法

直接在对话里提供语雀链接和 Token。

示例：

```text
请使用 yuque-frontend-requirements skill 处理这个语雀文档：
链接：https://www.yuque.com/owner/repo/doc-slug
token：你的语雀 Token
```

如果你希望输出更贴近现有项目，也可以补充参考上下文：

```text
请使用 yuque-frontend-requirements skill 处理这个语雀文档：
链接：https://www.yuque.com/owner/repo/doc-slug
token：你的语雀 Token

参考上下文：
- 日期选择组件参考：xxx/FormSearch.tsx 中的 RangePicker
- 查询按钮参考：xxx/AuditSearch.tsx
- 列表请求参考：xxx/index.tsx 中 updateTable 的写法
```

## 推荐输入格式

为了让输出更稳定，建议按下面格式提供信息：

```markdown
# 参考上下文
- 参考代码路径 1
- 参考代码路径 2
- 组件或接口调用约束

语雀链接：https://www.yuque.com/owner/repo/doc-slug
token：你的语雀 Token
```

如果链接不便直接提供，也可以给：

```text
repo namespace：owner/repo
doc slug：doc-slug
token：你的语雀 Token
```

## 处理流程

收到请求后，这个 Skill 会按以下顺序工作：

1. 解析语雀链接里的 `repo namespace` 和 `doc slug`
2. 使用语雀 API 拉取原始正文
3. 清洗语雀 HTML 为 Markdown，生成原始清洗文档
4. 从清洗后的内容里只提炼前端实现相关信息
5. 按固定模板输出前端技术实现需求文档

如果文档是私有的，但没有可用 Token，流程会在调用 API 前停止，并要求补充 Token。

## 会生成什么

通常会有两份 Markdown 产物：

1. `{语雀标题}-原文档.md`
2. `{语雀标题}-技术文档.md`

如果任务环境允许写文件，通常会优先放到项目根目录下的 `outputs/`。如果用户只需要对话输出，也可以直接在对话里返回技术文档内容。

例如：

- `outputs/增加检测维保长图展示入口-原文档.md`
- `outputs/增加检测维保长图展示入口-技术文档.md`

## 技术文档会怎么写

生成的技术文档会遵循这些规则：

1. `# 参考上下文`
   只承接你额外提供的实现锚点；如果你没有提供，这里会写 `暂无`。

2. `# 功能需求`
   会尽量保留原始需求标题，并改写成前端可执行描述，而不是简单复述原文。

3. `## 接口描述`
   只整理文档里已经明确给出的接口、参数、字段、示例；没写的部分会明确标记 `暂无`。

4. `## 工作流程`
   会固定保留“先理解需求，再拆分任务，再开发”的协作约束。

## 输出内容重点关注什么

这个 Skill 会优先提炼：

- 页面结构与模块布局
- 筛选区、表单、表格、分页、弹窗、抽屉
- 按钮行为、联动逻辑、显隐逻辑
- 字段定义、字段映射、状态枚举、默认值、校验规则
- 列表请求、详情请求、分页结构、响应字段
- 加载态、空态、失败兜底、权限控制
- 上传、预览、导出、复制、打开链接或 PDF 等前端行为

不会擅自补充：

- 原文里没有出现的接口字段
- 原文里没有出现的状态枚举
- 原文里没有出现的业务逻辑和条件分支

## 常见问题

### 1. 返回 404

一般有这几种原因：

- Token 没有私有知识库访问权限
- `repo namespace` 或 `doc slug` 不正确
- 请求没有带 `raw=1`
- 请求没有带 `User-Agent`

### 2. 生成的技术文档里很多地方是“暂无”

通常不是 Skill 出错，而是原文档没有提供足够信息。常见缺失项包括：

- 接口名称
- 请求路径
- 请求参数
- 响应字段
- 状态枚举
- 字段映射

这种情况下，Skill 会明确标记 `暂无` 或 `待确认`，不会擅自猜测。

### 3. 为什么“参考上下文”里没有语雀文档摘要

这是刻意设计的。`# 参考上下文` 只用于承接你额外提供的实现锚点，比如参考代码路径、组件写法、接口调用方式，而不是原文摘要。

### 4. 为什么有些编号点下面会出现短横线列表

当一个编号点下有多种状态、类型或分支逻辑时，会优先用短横线列表展开，方便阅读和后续开发拆分。

### 5. 我能只要“原文档”，不要“技术文档”吗

可以。你可以在需求里明确说明“只输出清洗后的 Markdown”，这样流程会停在原始清洗文档这一步。

## 一份完整示例

```markdown
# 参考上下文
- 日期选择组件：参考 sample-app/src/pages/xxx/FormSearch.tsx 中的 RangePicker
- 查询按钮：参考 sample-app/src/pages/xxx/AuditSearch.tsx
- 列表请求：参考 sample-app/src/pages/xxx/index.tsx 中 dispatch.xxx.updateTable

语雀链接：https://www.yuque.com/owner/repo/doc-slug
token：你的语雀 Token
```

## 给使用者的建议

1. 先确保自己能在浏览器中打开对应语雀文档。
2. 私有文档一定要确认 Token 权限。
3. 如果希望生成结果更贴近团队代码风格，尽量补充参考上下文。
4. 对生成结果里的 `待确认` 和 `暂无`，最好回头补齐产品或后端信息。
