# 语雀需求提炼

## 目标

收到语雀文档链接后，按以下目标执行：

1. 使用语雀 API 拉取文档内容。
2. 清洗语雀返回内容中的冗余 HTML 标签、样式和无关包裹结构，输出可读 Markdown。
3. 从 Markdown 中提炼与技术栈相关的信息（前端或后端）。
4. 按固定模板输出"技术实现需求文档"。

## 必要输入

- 语雀文档链接，或 `repo namespace + doc slug`
- 具备读取权限的语雀 Token
- 技术栈类型：`前端` 或 `后端`（默认根据需求内容判断，也可用户明确指定）

如果文档是私有库，且当前没有可用 Token，先向用户索要 Token 再继续。

## 执行流程

### 1. 解析文档标识

- 优先从语雀链接中解析 `repo namespace` 和 `doc slug`。
- 解析失败时，再要求用户补充 `repo namespace` 和 `doc slug`。

### 2. 拉取并清洗文档

- 运行 `scripts/yuque_to_markdown.py`。
- 用户给的是标准语雀链接时优先使用 `--url`。
- 链接不稳定或信息不完整时改用 `--repo-namespace` 和 `--doc-slug`。
- 通过 `--token` 或环境变量 `YUQUE_TOKEN` 传入 Token。
- 若用户没有指定输出文件名，使用语雀返回的 `data.title` 作为文件基名。
- 原始清洗结果文件命名规则：`{title}-原文档.md`
- 技术文档命名规则：`{title}-技术文档.md`

示例：

```bash
python3 scripts/yuque_to_markdown.py \
  --url "https://www.yuque.com/owner/repo/doc-slug" \
  --output-dir /tmp
```

支持的链接格式包括：

- `https://www.yuque.com/owner/repo/doc-slug`
- `https://tenant.yuque.com/owner/repo/doc-slug`

### 3. 判断技术栈并提炼信息

根据需求内容判断技术栈，或由用户明确指定：

**前端需求（JavaScript/TypeScript）**：优先关注页面、组件、交互、接口调用等前端实现相关内容。

**后端需求（Java）**：优先关注服务类、数据模型、接口定义、数据库设计、业务逻辑等后端实现相关内容。

### 4. 提炼后端(Java)相关信息

优先关注以下内容：

- 服务类、Controller、Service、Mapper/Dao 层级结构
- 接口路径、请求方法、请求参数、响应结构
- 业务逻辑处理流程、校验规则、异常处理
- 实体类、字段定义、数据类型、默认值
- 数据库表结构、关联关系、索引设计
- 枚举类、字典数据、状态流转
- 事务处理、缓存策略、日志记录
- 第三方服务调用、消息队列、定时任务
- 用户额外提供的参考代码路径或已有接口实现方式

不要虚构源文档中不存在的业务逻辑、接口字段或状态枚举。

### 5. 按模板输出技术实现文档

使用对应技术栈的模板作为输出骨架：

- 前端：`assets/frontend_requirement_template.md`
- 后端：`assets/backend_requirement_template.md`

并遵循以下规则：

- `# 参考上下文` 只承接用户额外明确提供的上下文，例如参考代码路径、已有接口、现有实现方式、包名类名命名规则。
- 用户没有提供额外上下文时，`# 参考上下文` 填写 `暂无`。
- 不要把原始语雀文档简单摘要后塞进 `# 参考上下文`。
- `# 功能需求` 下的二级标题必须尽量复用原始需求标题，不要统一写成"需求描述"。
- 每个原始需求标题下，输出的内容必须是"可落地的技术实现要求"，而不是复述原文。
- 对后端接口类需求，优先拆成：接口路径、请求参数、响应结构、业务逻辑、异常处理。
- 对后端数据类需求，优先拆成：表结构、字段定义、关联关系、索引设计。
- 对后端服务类需求，优先拆成：服务职责、业务流程、校验规则、事务处理。
- 如果用户已经提供接口文档，则在 `## 接口描述` 中按模板整理；如果只提供部分接口，只填写已知部分，其余写 `暂无`。
- 如果用户提供了"字段映射""状态枚举""字典类型""请求示例"等，优先保留原始命名，不自行改写。
- 如果某个结论无法从文档中确认，显式写明"待确认"或"暂无"，不要推断。
- 当某个编号条目下需要继续细分场景、类型或分支逻辑时，使用"编号项 + 次级短横线列表"的形式表达，不要把多个子点硬写在同一段里。
- 保留文末 `## 工作流程` 的协作约束。

## 输出质量要求

最终技术文档应满足：

- 仅输出 Markdown
- 保留原业务意图，但改写成技术可执行语言
- 按原始需求标题分组，不随意重组主题
- 缺失上下文、接口、字段、状态时明确写 `暂无`
- 对模糊项使用 `待确认`，不要补猜
- 能直接作为"先确认需求，再拆任务，再开发"的输入文档

## 参考示例中可复用的提炼模式

当用户给出类似下面这种完整提示词示例时，重点提炼这些稳定模式：

- `# 参考上下文` 里往往是"可借鉴代码路径"和"现有实现锚点"，应原样保留，不要压缩成一句话。
- `# 功能需求` 下的一级标题（如"列表需求"、"接口需求"）适合映射成最终文档中的独立需求段落。
- "接口路径、请求方法、请求参数、响应结构" 是后端接口最常见的拆分维度。
- "Service层逻辑、事务处理、异常处理" 是后端服务类需求的常见拆分维度。
- "表结构、字段定义、关联关系、索引" 是后端数据库设计的常见拆分维度。
- `## 工作流程` 不是功能点，而是协作约束，保留到文档末尾即可。

## 调试

如果语雀 API 返回 `404`，优先检查：

- Token 是否有私有知识库读取权限
- `repo namespace` 和 `doc slug` 是否完全正确
- 请求是否带了 `raw=1`
- 是否传了 `User-Agent`

详见 `references/yuque_api_usage.md`。
